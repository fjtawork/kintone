# kintone Clone プロジェクト概要

最終更新: 2026-02-15

## 1. 目的

- kintoneライクな業務アプリ作成基盤（ノーコード/ローコード）を提供する。
- 小規模企業が低コストで早く導入できることを優先する。
- 将来的に本家kintoneへ移行する企業も歓迎する。

## 2. 目指す提供価値

- 小さく始められる業務アプリ基盤を、買い切りで提供する。
- 長期ロックインより、必要に応じた移行しやすさを重視する。
- 導入初期に必要な機能を短期間で使えるようにする。

## 3. プロダクト方針（ビジネス）

- 提供形態はSaaS前提ではない。
- ビジネスモデルは買い切りライセンスを基本とする。
- アプリ内決済は実装しない。
- 決済や販売は外部チャネル（例: Shopify等）で実施し、購入後に納品する運用を想定する。

## 4. 想定ユーザー

- 初期フェーズの中小企業・小規模チーム。
- 自社で最低限の業務アプリを素早く構築したいユーザー。
- 将来的にkintone等へ移行する可能性があるユーザー。

## 5. スコープ（現時点）

- 業務アプリ管理（アプリ作成/一覧/設定）
- フィールド定義
- レコード登録/更新/一覧
- 権限・プロセス管理（JSONベース設定を含む）
- ユーザー管理
- 組織管理（部署・役職）

## 6. 非スコープ（現時点）

- サブスク課金
- アプリ内課金/請求管理
- マーケットプレイス機能
- 高度なテナント課金オペレーション

## 7. 移行前提の方針

- データ構造は可能な範囲で単純化し、移行用エクスポート/変換を実装しやすくする。
- 独自仕様への依存を最小化する（将来のkintone移行コストを抑える）。
- 移行時に必要なマッピング対象を明確化する。
  - アプリ定義（項目）
  - レコードデータ
  - ユーザー/組織関連の参照情報
  - 権限の差分

## 8. 今後の検討事項（事業・運用）

- 買い切りライセンス配布の運用手順（納品形式、更新配布、ライセンスキー有無）
- kintone移行エクスポート仕様（CSV/JSON、項目マッピングルール）
- 最小限の導入ドキュメント整備（管理者向けセットアップ手順）
